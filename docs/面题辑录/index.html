  • script标签：defer\async 作用

    [默认]: 同步加载和执行脚本，会阻塞HTML的解析

    [async]: 异步加载脚本，加载完成后，会立即执行脚本，并阻塞 HTML 的解析

    [defer]: 异步加载脚本，但会在 HTML 解析完成后执行，整个过程完全不会阻塞HTML的解析

    区别：
     (1) defer是在 HTML 解析完成后执行，async是在下载完成后立即执行
     (2) 多脚本时，defer会按照加载顺序执行，async则是哪个加载完成先执行哪个

    ps: 将延迟脚本放在页面底部仍是最佳选择

  • 事件三要素？事件的触发过程是什么，什么是事件代理？

    ✪ 三要素：事件源、事件类型、事件处理程序

    ✪ 事件触发过程有3个阶段：
       1. 捕获阶段：事件会从 window 往目标节点传播，所经过，遇到注册的捕获事件会触发
       2. 目标阶段：事件传递到目标节点，触发目标节点上的注册事件
       3. 冒泡阶段：从事件触发处往 window 传播，所经过，遇到注册的捕获事件会触发

    ps: 如果给同一个节点同时注册了捕获和冒泡事件，事件的触发会根据注册的顺序执行

    ✪ 事件代理：对于动态的子节点，将事件注册到其父节点上，实现事件代理，有效减少DOM绑定注销，节省内存

  • 闭包是什么？可以做什么？有什么问题？

    ✪ 定义：函数内部访问了其他作用域的变量，就是闭包。简单理解就是，函数返回了函数，并且返回的函数访问了外部函数的局部变量

    ✪ 好处
      1.变量一直维持在内存中，可以做缓存
      2.跨作用域访问时，存在局部作用域，有效提高访问速度
      3.避免全局声明变量，占用内存，局部声明，随用随回收

    ✪ 坏处
      1.存在内存泄漏问题，变量一直存在内存中增大了内存消耗。解决：使用后手动赋值null

    延伸应用：模块化、柯里化

    MDN: Object.hasOwn() 旨在取代 Object.prototype.hasOwnProperty()。

      其中官网就指出，babel-loader 很慢！Babel会将代码转为字符串生成AST，然后转变AST生成最终的代码，我们写的代码越多转换效率也就越低，那么我们就要优化loader的文件搜索范围，具体配置如下：
      1) 配置`exclude`，排除`node_modules`

       使用 HappyPack 发挥多核CPU电脑的威力，可以让webpack将任务分解成多个子进程同时执行，子进程处理完后再将结果发送给主进程。
    [tip: 由于js是单线程，所以要想发挥多核CPU的能力，只能通过多进程去实现，而无法通过多线程实现。]

          {
            test: /\.js$/

          }
        ]

      plugins: [
        new HappyPack({
          loaders: ["babel-loader"], //(2)
          threads: 4 //开启4个子进程
        })
      ]
    （ps: HappyPack作者不从事js工作 已停止维护，官方推出了thread-loader）

二、渲染模式​

  • SSR

    SSR也就是服务端渲染，页面在后端先获取到数据，然后发回前端注水渲染，如果你不是很熟悉，可以先看一下SSR相关的文章介绍。

  • SSG

    SSG 也就是静态站点生成，在构建时生成静态页面，不同用户访问到的都是同一个页面。

  • ISR

    SSG 的优点就是快，部署不需要服务器，任何静态服务空间都可以部署，而缺点也是因为静态，不能动态渲染，每添加一篇博客，就需要重新构建。所以有了ISR，增量静态生成，可以在一定时间后重新生成静态页面，不需要手动处理。

  • RSC

    Server component 是 React18 提供的能力， 与上面的 SSR 不同，相当于是流式的 SSR。

  • 图片(资源)预加载和懒加载

    ✪ 预加载
     [1.图片预加载]
      1) 借助css预加载：给样式设置背景图，并先隐藏
      2) 使用js脚本预加载：创建new Image()，并赋值src图片地址。图片较多时遍历处理
     [2.CSS预加载]
      1) 使用 link 标签，设置`rel`属性的值为`preload`，代表该资源必须预先获取和缓存
      2) 使用js动态创建 link 标签，设置相关属性（rel、as等）后插入head头部
     [3.JS预加载]
      同上。另外包括font字体等资源都可以
     [4.音视频文件预加载]
      audio 和 video 标签添加 `preload="auto"`

    扩展：一个最佳实践：将预加载的执行放入`window.onload`事件中，确保预加载时页面已经完全加载完成
     
    ✪ 懒加载
     1.计算dom在页面的位置信息，判断是否可视，执行加载
       缺点：比较消耗性能，需要监听滚动事件一直计算
       
     2.使用 `IntersectionObserver` 接口 可以"异步观察"节点的可视状态
       优点：异步监听，性能极佳，弥补了方案一的缺点，同时还能实现懒加载等业务功能
       扩展：还可以使用该Observer接口实现：无限滚动、监听元素可视后执行相关操作

  • 如何区分 preload 和 prefetch

    preload：是告诉浏览器页面必定需要的资源，在当前页面加载后，浏览器一定会加载这些资源并缓存
    prefetch：异步加载，是告诉浏览器其他页面可能需要的资源，浏览器不一定会使用这些资源

  • http1.1/http2/http3 之间的区别

    1. HTTP/1.1
      单一"请求-响应"模型：“1次1份” 一个请求对应一个响应
      缺点：队头阻塞，每个请求和响应的头部信息都在单独的数据包中发送，可能引起头部阻塞问题
      缺乏二进制帧：没有针对数据流的二进制帧，这使得性能优化复杂
      文本协议：HTTP/1.1使用明文文本进行通信，可以被窃听和篡改
      ps: 虽有管线化技术，但是很少使用，因为即使一次性请求多个，仍需要根据请求顺序排队响应，很可能造成其中一个丢包，下一个接口的数据给到这个接口了
      扩展：HTTPS的新增，在HTTP基础上新增了TLS，除TCP三次握手，TLS/1.2新增了四次握手，握手后才能加密通信；那么对于HTTP/1.1，越多的请求来回就意味着越多的未知数,缺点越加明显。

    2. HTTP/2
      多路复用，解决1.1队头阻塞问题（好比送外卖，老板聘请了多个外卖员进行送外卖，不用担心有多个外卖却只有一个外卖员导致送外卖效率很低）
      二进制帧：HTTP/2使用二进制帧格式，请求和响应报文都被划分为了各个不同的帧，分为首部帧和数据帧。还有个流标识符，使帧可以不用按顺序抵达，还能设置帧的优先级，标注流的权重。优点是：减少了数据的传输大小。
      服务器推送：服务器可以主动推送与客户端请求相关的资源，提升页面加载速度。缺点：不小心访问了也会把资源都加载了，可能造成DDoS非对称攻击，存在一个明显的"杠杆"，因此存在安全性问题。
      头部压缩：HTTP/2引入HPACK压缩算法，压缩报文首部，减少了数据传输的大小。
      
    3. HTTP/3
     传输层采用QUIC: HTTP/3使用基于UDP的传输协议QUIC，这减少了握手时间，提高了安全性
     多路复用和头部压缩：HTTP/3继承了HTTP/2的多路复用和头部压缩的特性，但是在QUIC协议的基础上更高效
     抗头部阻塞：HTTP/3解决了头部阻塞问题，可以更快地传送头部信息（数据帧的概念直接给了QUIC）
     QUIC数据包会添加一个Connection ID 即时wifi->4G IP变了，但是客户端和服务端协商好的连接没变，会识别为同一个连接，避免再次握手。

  • 浏览器输入网址(例: www.baidu.com)，回车后会发生什么？

    【基础流程】
    1. 首先，通过DNS对该网址进行解析，获取到IP地址
    2. 拿到IP地址，与服务器进行TCP连接（TCP三次握手）
    3. 建立TCP连接后，发起HTTP请求
    4. 服务器响应HTTP请求，浏览器获取html代码
    5. 浏览器解析html代码，并请求html代码中的资源（如css、js、图片等）
    6. 浏览器对页面进行渲染 呈现给用户
    7. 服务器结束TCP连接（TCP四次挥手）

    【扩展】
    1.DNS如何查找域名
      DNS通过递归查询，先去DNS缓存中，再去递归查找根域名服务器，然后去再下一级，如此递归查找后，找到了，给我们的web浏览器

    2.为什么HTTP协议要基于TCP来实现
      应用层HTTP协议是在传输层TCP协议之上的，TCP是一个端到端的可靠的面向连接的协议，无需担心数据传输等问题

    3.最后一步浏览器是如何对页面进行渲染的
     (1) 解析HTML文件，构成DOM树
     (2) 解析CSS文件，构成层叠样式表模型（CSSOM）
     (3) 将DOM和CSSOM合并为渲染树，代表一系列被渲染的对象
     (4) 计算渲染树各节点的大小位置等信息，被称之为 布局`layout`
     (5) 将渲染树的各个节点渲染到屏幕上，这步被称为 绘制`painting`

  • http缓存方式，强缓存、协商缓存、要说出具体的字段名及用法

    http缓存分为：强缓存 和 协商缓存
     · 强缓存：不用发请求到服务器，就拿到缓存文件的请求
     · 协商缓存：未命中强缓存，浏览器会向服务器发送资源加载请求。服务器判断浏览器本地资源是否失效，若可以使用，服务器不会返回资源，浏览器继续从缓存加载资源。
     · 无缓冲：协商缓存也未命中的话，服务器会将完整的资源返回给浏览器，浏览器加载新资源，并更新缓存
     
    ✪ 强缓存
     两个关键字段，expires 和 cache-control（优先级更高）
     expires：后面跟着资源的失效实践，这是一个`绝对时间` 缺点:就是当本地时间被修改，服务器和本地时间差异较大时，会导致缓存混乱，所以发展出了：Cache-Control
     cache-control：是一个`相对时间`，例：Cache-Control:max-age=3600 代表资源的有效时间相对本地时间是3600秒，由于是相对时间，所以服务器和本地有没有偏差也不会导致问题。
     
    ✪ 协商缓存
     两个关键字段，Last-Modify/If-Modify-Since 和 Etag/If-None-Match
     Last-Modify：代表资源的最后修改时间，缺点是修改时间因为精确到的是秒，所以可能存在资源更新了，时间却没变；还有可能资源的内容没有变化，但是时间更新了，导致无法使用缓存。于是出现了：Etag
     Etag：资源的唯一ID，只有资源变化了就会更新，解决了Last-Modify的问题
     第一次请求：响应返回 Last-Modify 和 Etag
     第二次请求：请求携带 If-Modify-Since 和 If-None-Match

    ps: 强缓存适用于静态资源，而协商缓存适用于需要频繁变动的资源。
        网站的缓存设置一般是：入口设置 no-cache 其他资源设置 max-age，这样入口文件会缓存但是每次都协商，保证能及时更新，而其他资源不发请求，减轻服务端压力。

    三级缓存原理
     1.先查找内存，如果内存中存在，从存在中加载
     2.如果内存中未查找到，选择磁盘获取，如果磁盘中有，从磁盘中加载
     3.如果磁盘中未查找到，那就进行网络请求
     4.加载到的资源缓存到磁盘和内容
